## STEP3 지식 추출 LLM 고도화 (2026-06-22)

- `pipeline.extract_knowledge`를 LLM 우선 + 휴리스틱 폴백 구조로 변경. `_llm_extract`가 `llm.generate_json`으로 knowledge 7항목·rules(4키)를 추출, 응답 불량 시 `_heuristic_extract`로 폴백. `extraction_mode`(llm|heuristic)를 리포트·API에 노출.
- **gemma4:latest는 추론 시 llama-server 크래시**(`GGML_ASSERT ... stack buffer overrun`, /api/generate·/api/chat 모두 500). 목록엔 보이나 사용 불가.
- 동작 확인 모델: `gemma4:31b-cloud`(✅, 채택), `qwen2.5:3b`·`qwen3:4b`(✅). → 기본 모델을 **gemma4:31b-cloud**로 변경(config.py).
- 테스트는 `tests/conftest.py`에서 `OLLAMA_HOST`를 도달 불가 주소로 설정해 휴리스틱 경로로 고정(클라우드 비의존, 결정론적). LLM 파싱 경로는 stub LLM 단위 테스트로 검증.

## 과제 앵글 7종 상한 — 억지 확장 지양 (2026-07-11)

3차 검증보고서 권고 #6은 "질문 템플릿 3종 → 8종 이상"을 제안했다. 앵글을 4→7종(설명/요약/규칙/용어/절차/사례/비교)으로 확장해 보고서가 예시한 사례형·비교형·절차형을 모두 포함시켰고, **8번째는 의도적으로 추가하지 않았다**.

- **근거**: 앵글은 세그먼트마다 LLM이 채우는데, 해당 내용이 없는 앵글은 빈 답변으로 드롭된다(예: 의안원문에는 현행↔개정 비교 소재가 적어 `compare`가 자주 빈다). 소재 없는 8번째 앵글을 억지로 넣으면 유효 행은 안 늘고 드롭·품질 필터 부하만 는다.
- **판단**: "8종 이상"은 숫자 목표가 아니라 다양성 확보가 목적이며, 7종으로 그 목적(질문 유형 다변화)은 달성됐다. 규모 확대는 앵글 수가 아니라 **소스 문서 확장(run_many·다중 업로드)**으로 푼다 — 실측 의안원문+검토보고서 결합 시 404행.
- 향후 새 앵글은 "빈 앵글 드롭률"을 확인해 실제 채워지는 경우에만 추가한다(패딩 금지 원칙과 동일).

## 의미 유사도 기준 현실화 0.95→0.75 (2026-07-11)

방법론 §공공기관 권장 기준의 의미 유사도 0.95는 **거의 원문 그대로**를 전제한 값이다. 본 파이프라인은 프롬프트에서 "원문을 통째로 베끼지 말되 용어·수치·조문은 보존하고 표현만 새로 쓰라"고 지시해 **의도적으로 재진술**한다. 그래서 output↔원문 표면 임베딩 유사도는 자연히 낮다.

- **실측**(SEMANTIC_ENABLED=1, ko-sroberta, 2024598 전수 67행): 의미 유사도 **0.799**. 같은 데이터의 엔티티 근거성 0.988·환각 1.5%로 **사실 보존은 별도 지표로 검증**됨.
- **판단**: 0.799는 품질 저하가 아니라 재진술 설계의 결과다. 문장 임베딩 코사인 0.7~0.75는 통상 '재진술 등가(같은 의미)'로 본다. `std_semantic` 기본값을 **0.75**로 현실화(env STD_SEMANTIC로 조정 가능). 원문 복사에 가깝게 만들어 0.95를 맞추는 것(충실도 상향)은 다양성·학습 가치를 떨어뜨려 채택하지 않음.
- 사실 왜곡은 의미유사도가 아니라 엔티티 근거성·환각율·부정문 반전 플래그로 잡는다(역할 분리).
