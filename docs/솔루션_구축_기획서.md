## 1. 추진 배경 및 개요

### 1.1 배경

공공기관은 규정·지침·매뉴얼·민원 FAQ 등 방대한 업무 문서를 보유하고 있으나, 대부분 비정형 상태로 존재해 LLM·RAG·AI Agent 학습에 즉시 활용하기 어렵다. 민간에서는 Unsloth Studio처럼 "문서 업로드 → 데이터셋 자동 생성 → 파인튜닝"을 하나의 웹 UI로 처리하는 노코드 플랫폼이 등장했으나, 이를 공공기관에 그대로 도입하기에는 다음 제약이 있다.

- **보안·망분리.** 공공기관 내부망은 외부 인터넷과 분리(망분리)되어 있어, 클라우드 API·외부 모델 다운로드에 의존하는 구조를 쓸 수 없다.
- **개인정보·민감정보.** 업무 문서에 포함된 개인정보·내부정보가 외부로 유출되어서는 안 되며, 전 과정이 기관 내부에서 처리되어야 한다.
- **자원 제약.** 고성능 GPU 클러스터를 전제로 한 대규모 파인튜닝 인프라를 모든 기관이 갖추기는 어렵다.
- **한글 문서.** HWP 등 국내 공공 문서 포맷 처리가 필수다.

### 1.2 목적

Unsloth Studio의 핵심 가치(노코드 데이터셋 생성·검증, 오프라인 동작, 표준 학습 포맷 출력)는 계승하되, **불필요한 기능을 덜어내고 공공기관 환경에 최적화한 경량 데이터셋 생성·검증 플랫폼**을 구축한다. 본 시스템은 PRD/TRD에 정의된 8단계 파이프라인을 제품화하여, 기관 실무자가 문서를 올리면 검증된 학습용 데이터셋(CSV/JSON/JSONL)을 받을 수 있게 한다.

### 1.3 경량화 정의 (무엇을 덜어내는가)

본 사업의 "경량화"는 Unsloth Studio 대비 다음과 같이 범위를 좁히는 것을 의미한다.

| 구분 | Unsloth Studio | 본 경량 시스템 |
| --- | --- | --- |
| 핵심 책임 | 데이터 생성 + **파인튜닝 학습 실행** + 추론 서빙 | **데이터 생성·검증까지** (학습 실행은 위임) |
| 모델 범위 | 500+ 모델 학습/추론 | 데이터 생성용 소형 LLM 1~2종(로컬) |
| UI 복잡도 | 학습·아레나·옵저버빌리티 등 풀 기능 | 업로드 / 생성 / 검증 / 다운로드 4단계 중심 |
| 인프라 | 고사양 GPU 전제 | CPU/소형 GPU에서 동작, GPU는 선택 |
| 배포 | 로컬 + 통합 환경 | **완전 온프레미스·망분리 내부망** |

> **핵심 메시지**
> "학습을 쉽게 하는 도구(Unsloth)"가 아니라 "**학습 데이터를 안전하게 잘 만드는 도구**"에 집중한다. 학습 실행 자체는 PRD Non-Goals대로 Unsloth 등 외부 도구에 위임하고, 본 시스템은 Unsloth가 바로 소비 가능한 포맷까지 책임진다.

## 2. 현황 분석 (As-Is)

### 2.1 기존 자산

사내에는 이미 PRD/TRD에 따라 8단계 파이프라인이 구현되어 있고 end-to-end 테스트를 통과한 상태다(Python + FastAPI 웹 UI + 로컬 Ollama, 미가용 시 mock 폴백 + 자체 검증 루프). 즉 **데이터 생성의 핵심 절반은 이미 확보**되어 있으며, 본 사업은 이를 공공기관용 제품으로 고도화·경량화·보안강화하는 작업이다.

### 2.2 Unsloth Studio 벤치마크 요약

| Studio 기능 | 본 시스템 채택 여부 | 비고 |
| --- | --- | --- |
| Data Recipes(문서→합성 데이터셋) | **채택(경량)** | 자체 8단계 파이프라인으로 구현 |
| 노코드 학습(LoRA/QLoRA) | 제외(위임) | 데이터셋만 산출, 학습은 Unsloth로 |
| 옵저버빌리티(학습 모니터링) | 축소 채택 | 생성·검증 진행률 모니터링으로 대체 |
| Model Arena(모델 비교) | 선택(차기) | 데이터 품질 A/B 비교로 응용 가능 |
| 내보내기(GGUF/safetensors) | 변형 채택 | 학습 포맷 JSONL 내보내기로 대체 |
| 100% 오프라인·로컬 | **필수 채택** | 공공기관 핵심 요건 |
| 토큰 기반 인증(JWT) | **채택·강화** | 기관 SSO/계정 연계 |

## 3. 목표 시스템 (To-Be)

### 3.1 비전

> 망분리된 기관 내부망에서, 실무자가 업무 문서를 올리면 **수작업 없이 검증된 AI 학습 데이터셋**을 생성하는 안전한 노코드 플랫폼.

### 3.2 핵심 목표

1. PRD 8단계 파이프라인을 제품화하여 문서 → 데이터셋 자동화율 90% 이상 달성.
2. 전 과정 온프레미스·오프라인 동작(외부 통신 0건).
3. Instruction·Q&A·RAG 3종 + Unsloth 학습 포맷 4종(Raw/Alpaca/ShareGPT/ChatML) 출력.
4. 검증 우선 루프(Validate → Preview → Refine → Run)로 품질 점수 90점 이상, PASS 95% 이상.
5. CPU 또는 소형 GPU 환경에서 동작하는 경량 구성.

### 3.3 비목표 (Non-Goals)

- 모델 파인튜닝(학습) 실행 — Unsloth/외부 도구에 위임.
- 실시간 협업 편집·문서 작성 도구.
- 대규모 멀티 GPU 학습 인프라 운영.

## 4. 기능 범위

PRD 기능요구(FR-01~19)를 공공기관 경량화 관점에서 3단계 우선순위로 재배치한다.

### 4.1 1단계 (MVP, 필수)

| ID | 기능 | 경량화 메모 |
| --- | --- | --- |
| FR-01 | 다중 포맷 업로드(PDF/DOCX/XLSX/HWP/TXT/MD/이미지) | HWP·OCR 국내 포맷 우선 |
| FR-02/03 | 문서 분석 + 도메인 자동 분류 | 공공행정 도메인 우선 튜닝 |
| FR-04/05 | 전문가 Agent 라우팅 + 지식 추출 | 공공·법률·금융 매핑 기본 제공 |
| FR-06 | Instruction/Q&A/RAG 3종 생성 | 핵심 산출물 |
| FR-07/08 | CSV/JSON 산출 | 표준 스키마 고정 |
| FR-09 | Unsloth 학습 포맷 변환(4종) | 학습 도구 접점 |
| FR-12/13/14 | Validator + LLM Judge + 검증 루프 | 품질 게이트 |
| FR-15 | 크기·구조 검증(최소 100행) | Run 차단 게이트 |

### 4.2 2단계 (고도화)

FR-10 채팅 템플릿 적용, FR-11 합성 데이터 증강, FR-16 리포트 생성, FR-17 버전 관리.

### 4.3 3단계 (확장/선택)

FR-18 RAG 백엔드 연동(ChromaDB/FAISS 우선, 온프레미스 가능 제품만), FR-19 HF 게시(**망분리 환경에서는 보류**, 내부 저장소로 대체).

## 5. 시스템 아키텍처

### 5.1 구성도 (논리)

```
[기관 내부망 / 망분리 영역]

  실무자 PC ──(내부망 HTTPS)──► 웹 UI (FastAPI + 정적 프런트)
                                      │
                                      ▼
                          ┌─────────────────────────┐
                          │   8단계 파이프라인 엔진    │
                          │  (runner.py 오케스트레이션)│
                          └───────────┬─────────────┘
                                      │
        ┌─────────────┬──────────────┼───────────────┬──────────────┐
        ▼             ▼              ▼               ▼              ▼
  Document Loader  Analyzer/    Knowledge      Dataset Gen +   Validator +
  (HWP/PDF/OCR)   Classifier    Extractor      Format Conv.    LLM Judge
        │             │              │               │              │
        └─────────────┴──────────────┴───────────────┴──────────────┘
                                      │
                              로컬 LLM 백엔드
                          (Ollama / llama.cpp, GGUF)
                                      │
                                      ▼
                     산출물 저장소(파일시스템/내부 DB)
                csv · json · unsloth jsonl · dataset_report.md
```

전 구성요소는 외부 인터넷 접속 없이 기관 내부망에서 동작한다.

### 5.2 8단계 파이프라인 (TRD 계승)

STEP1 문서 분석 → STEP2 전문가 Agent 라우팅 → STEP3 지식 추출 → STEP4 데이터셋 생성 → STEP4.5 Unsloth 포맷 변환 → STEP5 CSV → STEP6 JSON → STEP7 품질 검증(검증 루프) → STEP8 버전 관리. 단계 간 데이터 계약(필드명·컬럼명)은 TRD §4 고정값을 그대로 준수한다.

### 5.3 컴포넌트-기존 코드 매핑

| 컴포넌트 | 기존 코드 | 경량화 작업 |
| --- | --- | --- |
| 오케스트레이션 | `src/runner.py` | 진행률 이벤트 추가 |
| STEP1~4.5 | `src/pipeline.py` | 공공 도메인 프롬프트 보강 |
| 검증 루프 | `src/validate.py` | 게이트 임계값 설정화 |
| 합성 증강 | `src/synthetic.py` | 다양성·밸런싱 점검 |
| 직렬화 | `src/export.py` | JSONL 학습 포맷 export 강화 |
| 로더 | `src/loaders.py` | HWP/OCR 어댑터 추가 |
| LLM 호출 | `src/llm.py` | 로컬 모델 고정, mock 폴백 유지 |
| 웹 UI | `web/app.py` + `index.html` | 4단계 위저드 UX |

### 5.4 경량 LLM 백엔드 전략

- **기본:** 로컬 소형 instruct 모델(예: 3B~8B급) 1종을 GGUF로 탑재, Ollama/llama.cpp로 추론. 데이터 생성·합성·LLM Judge 모두 이 모델로 처리.
- **폴백:** 모델 미탑재/장애 시 mock 경로로 파이프라인이 끝까지 동작(테스트 가능성·시연 안정성 유지).
- **정밀도:** CPU 환경은 4비트 양자화, 소형 GPU(8GB+) 환경은 16비트 옵션 제공.
- **학습·서빙 정밀도 일치 원칙**은 학습 단계(외부)에서 적용하도록 가이드 문서로 안내.

## 7. 보안 및 컴플라이언스 (공공기관 핵심)

| 영역 | 요구사항 | 구현 방안 |
| --- | --- | --- |
| 망분리 | 외부 통신 차단 | 오프라인 설치, 외부 호출 코드 경로 제거·차단 |
| 데이터 주권 | 문서·데이터셋 외부 유출 금지 | 전 과정 내부 처리, 임시파일 암호화·자동 삭제 |
| 인증·권한 | 사용자 식별·접근통제 | JWT + 기관 SSO/계정 연계, 역할별 권한 |
| 개인정보 | PII 비식별화 | 업로드 시 마스킹·비식별 옵션, 로그 최소화 |
| 추적성 | 원문 출처 보존 | `source_document` 전 레코드 보존, 감사 로그 |
| 라이선스 | 오픈소스 의무 검토 | Unsloth Studio UI는 AGPL-3.0 → 코드 차용 시 검토, 자체 구현 우선 |
| 모델 라이선스 | 상업·공공 사용 조건 | 탑재 모델 라이선스 사전 검토(공공 활용 가능 모델 선정) |

> **주의:** Unsloth **코어**는 Apache 2.0이나 **Studio UI 등 일부 구성요소는 AGPL-3.0**이다. 본 시스템 UI는 Studio 코드를 직접 차용하지 않고 자체 구현하여 라이선스 리스크를 회피한다(기능을 벤치마크할 뿐 코드를 복제하지 않음).

| 3. 검증 고도화 | M4 | 검증 루프·게이트, 합성 증강, 리포트·버전 관리 |
| 4. 보안·이행 | M5 | 망분리 패키징, 인증·감사, 보안성 검토 대응 |
| 5. 시범 운영 | M6 | 파일럿(공공행정 도메인), KPI 측정, 안정화 |

> Now/Next/Later 관점: **Now** — MVP(데이터 생성+검증+학습 포맷 export). **Next** — 합성 증강·리포트·버전 관리, 도메인 확장. **Later** — RAG 백엔드(온프레미스) 연동, 품질 A/B 비교.

## 11. 위험 및 대응

| 위험 | 영향 | 대응 |
| --- | --- | --- |
| HWP·스캔 문서 파싱 정확도 | 데이터 품질 저하 | 전용 파서 + OCR 보정, 파싱 실패 행 검증 제외 |
| 경량 모델 생성 품질 한계 | 점수 미달 | LLM Judge 게이트 + 합성 증강 + 프롬프트 튜닝 |
| 망분리 내 패키지·모델 반입 | 구축 지연 | 오프라인 설치 패키지 사전 구성·검증 |
| 오픈소스 라이선스(AGPL) | 법적 리스크 | UI 자체 구현, 모델 라이선스 사전 검토 |
| GPU 부재 환경 성능 | 처리 지연 | 4비트 양자화·배치 처리·진행률 안내 |

## 12. 기대 효과

- **업무 효율:** 문서→데이터셋 수작업 제거로 데이터 구축 기간 대폭 단축.
- **보안 준수:** 전 과정 내부망 처리로 개인정보·내부정보 유출 위험 차단.
- **재사용성:** 표준·검증된 데이터셋으로 기관 내 다양한 AI 사업(RAG·챗봇·Agent)에 재활용.
- **확장성:** 도메인·포맷·RAG 백엔드를 매핑/어댑터 추가만으로 확장.
- **자립성:** 외부 클라우드 의존 없이 기관이 직접 운영·관리.

## 13. 결정 필요 사항 (Decisions Needed)

TRD §9 항목 중 본 사업에서 우선 확정해야 할 사항:

- 탑재 로컬 LLM 모델 선정(공공 활용 가능 라이선스 + 한국어 성능).
- HWP/OCR 파서 라이브러리 확정.
- 검증 루프 자체 구현 범위(Data Recipes 위임 불가 — 망분리).
- 채팅 템플릿 기본 대상 모델.
- 데이터셋 저장·버전 관리 방식(파일시스템 vs 내부 DB).
- 인증 연계 방식(기관 SSO 종류).
