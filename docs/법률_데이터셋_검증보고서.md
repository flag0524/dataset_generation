## 2. 결함 상세

### 2-1. ❌ question 필드: 기계적 절단 생성 (106/108건)

question이 **input의 앞부분을 잘라 접미 템플릿을 붙이는 방식**으로 생성되었습니다.

접미 템플릿 4종 × 각 27건:
- `…내용을 설명해 줄 수 있나요?`
- `…을(를) 요약하면 무엇인가요?`
- `…에 적용되는 처리 기준은 무엇인가요?`
- `…핵심 용어는 무엇인가요?`

이로 인한 파생 결함:
- **괄호 미닫힘 9건**: `"…일부개정법률안 (윤관석의원 대표발의 내용을 설명해 줄 수 있나요?"` — 문장 중간 절단.
- **조사 미처리 27건**: `을(를)` 그대로 노출.
- **청크 번호 유출 16건**: `"4 - 제99조에…"` 처럼 원문 청킹 인덱스가 질문에 포함.
- **의미 불성립 질문**: 예) id 102 `"(현행과 같음) <신 설> 8의2.을(를) 요약하면 무엇인가요?"`

### 2-2. ❌ chatml/sharegpt 학습 설계 결함: 원문 컨텍스트 소실

chatml·sharegpt의 user 턴에는 위 **절단형 question만** 들어가고, JSON/alpaca의 `input`(법안 원문 청크)이 **전혀 포함되지 않습니다.** system 메시지도 0건입니다.

- 이대로 학습하면 모델은 "잘린 법안 제목 → 답변"을 **암기**하도록 훈련됨. RAG 파이프라인용(문서 근거 기반 응답) 학습 목적과 정면 충돌.
- alpaca 포맷은 `input`에 원문이 있어 정상이지만, 대화형 두 포맷은 재생성 필요.

### 2-3. ❌ keyword 메타데이터 무효

108건 전원이 동일한 `["공사대금", "수급인", "하수급인"]`. 문서 레벨 키워드가 레코드 전체에 복사된 것으로, 레코드별 검색·필터링 용도로 사용 불가.

### 2-4. ⚠️ 저품질 청크 유입

- `"(현행과 같음) <신 설> 8의2."`(19자) 같은 신구조문 대비표 파편이 청크로 유입되어 4개 레코드 생성. 그 결과 id 102의 output은 `"기존 내용과 동일한 조항이 새롭게 신설되었습니다."` — 의미상 모순(사실상 환각).
- `"제38조의4를 다음과 같이 신설한다."`처럼 후속 조문 내용이 잘려나간 청크도 존재 → output이 `"제38조의4 조항을 신설함."`(15자)으로 정보량 0에 가까움.

### 2-5. ⚠️ 다양성 부족

- 소스 문서 **1개**(2024590 건설산업기본법 개정안), category **knowledge 단일**, domain 단일, created_date 단일.
- 108건 = 27청크 × 4태스크 구조. 실질 정보 단위는 27개. 파인튜닝 데이터셋으로는 규모·다양성 모두 부족하며, 특정 법안 표현에 과적합 위험.

## 3. 권고 조치 (우선순위순)

1. **question 생성기를 LLM 기반으로 교체** — input 앞부분 절단 방식 폐기. 최소한 (a) 괄호/문장 경계 검사, (b) 조사 처리, (c) `^\d+\s*-\s` 접두어 제거 필터를 생성 파이프라인에 추가.
2. **chatml/sharegpt 재생성** — user 턴에 원문 청크를 포함하거나(`"다음 내용을 …: {input}"` 형태), system 메시지로 instruction을 넣는 구조로 변경. 현재 형태로는 학습 투입 금지 권장.
3. **keyword를 레코드(청크) 단위로 추출** — 현재 문서 단위 복사 로직 수정.
4. **청크 품질 게이트 추가** — 최소 길이(예: 50자 이상), 신구조문 대비표 파편(`(현행과 같음)`, `<신 설>` 등) 제외 규칙.
5. **소스 문서 확대** — 의안 다수 + 자치법규(ELIS) 등으로 확장해 27청크 의존 구조 해소. category도 knowledge 외 유형 추가.
