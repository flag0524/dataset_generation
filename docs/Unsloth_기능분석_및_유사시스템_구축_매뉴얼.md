## 1. 문서 개요

**목적.** 이 문서는 LLM 파인튜닝 플랫폼인 **Unsloth** 및 그 웹 UI인 **Unsloth Studio**의 전체 기능을 정리하고, 이와 유사한 도메인 특화 데이터셋 생성·학습 시스템을 자체 구축할 때 알아야 할 핵심 개념·구성요소·기술 요건을 매뉴얼 형태로 제공한다.

**대상 독자.** 기획·의사결정자, 개발·엔지니어, 운영 담당자를 포함한 실무팀 전체. 1~3장은 개념 중심, 4~6장은 구축·기술 요건 중심, 7~8장은 운영·체크리스트로 구성된다.

**범위.** Unsloth는 파인튜닝(모델 학습) 도구이고, 진행 중인 사내 프로젝트는 문서에서 데이터셋을 생성하는 도구다. 두 시스템은 데이터셋 준비 단계에서 직접 맞닿아 있으므로, 본 매뉴얼은 Unsloth의 데이터 파이프라인(Data Recipes)과 사내 8단계 파이프라인의 접점을 특히 강조한다.

> **한눈에 보기**
> - **Unsloth =** 오픈소스 LLM 파인튜닝 엔진 + 노코드 웹 UI(Studio). 학습 속도 2배, VRAM 70% 절감.
> - **우리 시스템 =** 업무 문서를 입력받아 LLM/RAG 학습용 데이터셋(CSV/JSON/JSONL)을 만드는 8단계 파이프라인.
> - **접점 =** 우리가 만든 데이터셋이 Unsloth 같은 도구의 학습 입력이 된다. 따라서 출력 포맷 호환성이 핵심.

## 2. Unsloth란 무엇인가

Unsloth는 LLaMA, Mistral, Qwen, Gemma, gpt-oss 등 오픈 LLM을 **빠르고 적은 메모리로 파인튜닝(추가 학습)**할 수 있게 해 주는 오픈소스 라이브러리다. 복잡한 연산을 자체 GPU 커널로 변환해 Hugging Face 대비 학습을 2~5배 가속하고, 정확도 손실 없이 VRAM 사용량을 크게 줄인다. 무료 Colab/Kaggle 노트북에서 최소 3GB VRAM으로도 학습이 가능하다.

### 2.1 파인튜닝이란

사전학습된 모델의 동작을 특정 도메인·작업에 맞게 재학습하는 과정이다. 주요 효과는 다음과 같다.

- **새 지식 주입 —** 도메인 특화 정보를 모델 가중치에 학습시킨다.
- **행동 커스터마이즈 —** 말투·성격·응답 스타일을 조정한다.
- **작업 최적화 —** 특정 업무(분류·요약·계약 분석 등)에서 정확도를 높인다.

> **파인튜닝 vs RAG**
> RAG는 추론 시점에 외부 문서를 참고할 뿐 모델 가중치를 바꾸지 않는다. 파인튜닝은 가중치 자체를 학습시키므로 RAG의 기능을 대체·포함할 수 있다. 실무에서는 둘을 조합하는 것이 일반적이다(고정 지식은 파인튜닝, 자주 바뀌는 데이터는 RAG).

### 2.2 학습 방식 (LoRA / QLoRA / FFT)

| 방식 | 설명 | 메모리 | 권장 상황 |
|------|------|--------|-----------|
| LoRA | 원본 가중치는 동결, 얇은 어댑터(A·B 행렬)만 16비트로 학습 | 중간 | 정확도 우선, 표준 SFT |
| QLoRA | LoRA + 4비트 양자화. 큰 모델을 최소 자원으로 | 최저(9B를 6.5GB) | 입문·자원 제약 (권장 시작점) |
| FFT(전체) | 모든 가중치 학습. 고비용 | 최고 | 특수한 경우만. 보통 불필요 |

**권장 흐름.** QLoRA → (필요 시) LoRA → (정말 필요할 때만) FFT. LoRA에서 안 되는 것이 FFT에서 마법처럼 되지는 않는다.

### 2.3 지원 학습 유형

SFT(지도학습) 외에도 선호 최적화(DPO·ORPO·KTO), 강화학습(GRPO·GSPO), 지속 사전학습, 비전·멀티모달, 음성합성(TTS), 임베딩, BERT 계열까지 폭넓게 지원한다.

## 3. Unsloth Studio — 노코드 웹 UI

2026년 3월 출시된 **Unsloth Studio**는 학습·실행·내보내기를 하나의 **로컬 웹 UI**에서 처리하는 오픈소스 노코드 도구다. 500개 이상 모델을 2배 빠르게, VRAM 70% 절감으로 학습한다. 100% 오프라인·로컬 동작이 가능하며 사용 텔레메트리를 수집하지 않는다.

### 3.1 핵심 기능

| 기능 | 설명 |
|------|------|
| 로컬 모델 실행 | GGUF·safetensor 모델 검색·실행. 자가치유 툴콜링, 웹검색, 코드 실행(Bash·Python) 내장 |
| 노코드 학습 | PDF·CSV·JSON·YAML 업로드 후 즉시 학습. LoRA·FP8·FFT·PT 커널 자동 최적화 |
| Data Recipes | 문서를 그래프-노드 워크플로로 데이터셋(합성 포함) 변환. NVIDIA NeMo Data Designer 기반 |
| 옵저버빌리티 | 학습 손실·그래디언트 노름·GPU 사용률 실시간 추적. 모바일에서도 모니터링 |
| Model Arena | 두 모델(예: 베이스 vs 파인튜닝)을 나란히 비교 |
| 내보내기/저장 | safetensors·GGUF로 내보내 llama.cpp·vLLM·Ollama·LM Studio에서 사용. 학습 이력 저장 |
| API 엔드포인트 | OpenAI 호환 API. Claude Code·Codex 등에서 로컬 LLM 사용 |
| 프라이버시/보안 | 100% 오프라인, 토큰 기반 인증(암호화 비밀번호 + JWT) |

### 3.2 표준 워크플로우

1. Studio 실행 (설치 후 `unsloth studio` 명령)
2. 로컬 파일 또는 연동에서 모델 로드
3. PDF·CSV·JSONL에서 학습 데이터 가져오기 또는 처음부터 구성
4. Data Recipes에서 데이터 정제·확장·합성
5. 권장 프리셋 또는 커스텀 설정으로 학습 시작
6. 학습 모델과 대화하고 베이스 모델과 출력 비교
7. 사용 중인 스택으로 로컬 저장·내보내기

### 3.3 설치 및 라이선스 (참고)

- **macOS/Linux/WSL:** `curl -fsSL https://unsloth.ai/install.sh | sh`
- **Windows:** `irm https://unsloth.ai/install.ps1 | iex`
- **실행:** `unsloth studio -H 0.0.0.0 -p 8888`  (Docker 이미지 `unsloth/unsloth` 제공)

**라이선스.** 코어 패키지는 Apache 2.0, Studio UI 등 일부 구성요소는 AGPL-3.0의 이중 라이선스 구조다. 사내 제품에 코드를 차용할 경우 AGPL 의무(소스 공개)를 반드시 검토해야 한다.

## 4. 유사 시스템 구축 — 알아야 할 핵심 개념

Unsloth 같은 데이터에서 학습, 배포로 이어지는 파이프라인을 자체 구축하려면 다음 다섯 축을 이해해야 한다.

| 축 | 핵심 질문 | 관련 결정 |
|------|-----------|-----------|
| 데이터 | 무엇을, 어떤 포맷으로 학습하나 | 수집·정제·합성·스키마 |
| 모델 | 어떤 베이스 모델을 쓰나 | 크기·instruct/base·라이선스 |
| 학습 | 어떻게 학습하나 | LoRA/QLoRA, 하이퍼파라미터 |
| 평가 | 잘 됐는지 어떻게 아나 | 검증셋·자동/수동 평가 |
| 배포 | 어디서 어떻게 서빙하나 | GGUF/vLLM, API, 추론 엔진 |

### 4.1 데이터셋이 결과를 좌우한다

Unsloth 가이드의 핵심 메시지는 데이터 품질과 양이 파인튜닝 결과를 거의 결정한다는 것이다. 일반적으로 **질문-답변(QA) 쌍** 형태로 잘 구조화된 데이터가 가장 효과적이다. 단순 문서 덤프는 (코드 학습 등 일부 예외를 빼면) 효과가 떨어진다. 이는 사내 시스템의 설계 원칙(단순 변환이 아니라 문서 이해, 지식 구조화, 학습 데이터 변환)과 정확히 일치한다.

#### 주요 데이터 포맷

| 포맷 | 구조 | 용도 |
|------|------|------|
| Instruction (Alpaca) | instruction / input / output | 지시 따르기 학습 |
| Q&A (대화형) | question / answer / source | 챗·도메인 QA |
| ChatML/ShareGPT | role 기반 멀티턴 메시지 | instruct 모델 대화 학습 |
| RAG 레코드 | id / title / content / metadata | 검색 증강용 인덱싱 |

> **사내 시스템과의 정합성**
> 현재 8단계 파이프라인은 Instruction·Q&A·RAG 세 형태를 동시에 생성하도록 설계되어 있어 Unsloth 권장 포맷과 호환된다. 추가로 **Unsloth/HF가 바로 읽는 JSONL(메시지 배열) 내보내기**를 지원하면 학습 도구와의 접점이 완성된다.

### 4.2 합성 데이터 생성 (Data Recipes의 핵심)

Unsloth의 합성 데이터 노트북과 Studio의 Data Recipes는 PDF·동영상 등 비정형 문서를 자동 파싱해 QA 쌍을 생성하고, 로컬 모델(예: Llama 3.2)로 자동 정제한다. 유사 시스템에 반드시 갖춰야 할 요소는 다음과 같다.

- **문서 파서 —** PDF·DOCX·XLSX·HWP·PPT·TXT·MD·OCR 이미지 로더
- **청킹/구조화 —** 문서를 의미 단위로 분할하고 용어·규칙·프로세스 추출
- **QA 생성기 —** LLM으로 question/answer 쌍 생성 (도메인 전문가 프롬프트)
- **자동 정제 —** 중복 제거, 원문 근거 확인, 품질 점수화
- **증강 —** 분량 부족 시 합성 데이터로 최소 행 수 충족

### 4.3 모델·하이퍼파라미터 이해

입문자는 작은 instruct 모델(예: Llama 3.1 8B)로 시작하는 것이 권장된다. instruct 모델은 ChatML/ShareGPT 같은 대화 템플릿으로 바로 학습 가능하고 base 모델보다 적은 데이터로 충분하다. 학습·서빙을 같은 정밀도로 맞추면 정확도가 보존된다(4비트로 서빙하려면 4비트로 학습).

| 하이퍼파라미터 | 의미 | 참고값 |
|------|------|--------|
| max_seq_length | 컨텍스트 길이 | 테스트 2048 |
| learning_rate | 학습률(낮을수록 정밀·느림) | 2e-4 → 5e-5 |
| num_train_epochs | 전체 반복 횟수 | 1~3 (과적합 주의) |
| batch / grad_accum | 배치·그래디언트 누적 | 2 / 4 |
| loss(관찰) | 학습 손실 | 0.5~1.0 양호, 0은 과적합 |

## 5. 시스템 아키텍처와 기술 스택

### 5.1 권장 컴포넌트 구성

| 계층 | 역할 | 기술 옵션 |
|------|------|-----------|
| 입력/로더 | 다양한 문서 포맷 적재 | Python 로더, OCR(Tesseract), pdfplumber |
| 처리 파이프라인 | 분석, 추출, 생성, 검증 | FastAPI 오케스트레이션 (사내 runner.py) |
| LLM 추론 | 생성·판정·정제 | 로컬 Ollama / vLLM / OpenAI·Anthropic API |
| 데이터 저장 | 데이터셋·메타데이터 | CSV·JSON·JSONL, (선택) DB |
| 검증 루프 | 품질·중복·근거 게이트 | Validator + LLM Judge |
| 내보내기 | 학습 도구 호환 산출 | CSV / JSON / Unsloth JSONL / 리포트 |
| 웹 UI | 업로드·생성·다운로드 | FastAPI + 정적 프런트(사내 web/app.py) |
| RAG 백엔드(확장) | 벡터 검색 | ChromaDB·FAISS·Milvus·Pinecone |
| 파인튜닝(확장) | 실제 모델 학습 | Unsloth·OpenAI·LLaMA·Mistral |

> **사내 프로젝트 매핑**
> 위 구성은 이미 사내 8단계 파이프라인(runner.py·pipeline.py·validate.py·synthetic.py·export.py·loaders.py·llm.py + web/app.py)과 대응한다. 즉 Unsloth 유사 시스템의 데이터 생성 절반은 이미 구현되어 있고, 남은 절반은 **학습·배포 연동**이다.

### 5.2 데이터 생성 파이프라인 (8단계)

Unsloth가 데이터를 입력으로 가정하는 반면, 사내 시스템은 그 입력을 만드는 데 특화되어 있다. 8단계는 다음과 같다.

- **1. 문서 분석 —** 제목·목적·분야·용어·프로세스·규칙·FAQ 추출, 메타데이터 생성
- **2. 도메인 전문가 선택 —** 문서 유형에 따라 전문가 역할 자동 분기(공공·법률·금융 등)
- **3. 지식 추출 —** 정의·목적·절차·조직·입출력·규칙 추출
- **4. 데이터셋 생성 —** Instruction·Q&A·RAG 세 형태 동시 생성
- **5. CSV 생성 —** domain_dataset.csv (표준 컬럼)
- **6. JSON 생성 —** domain_dataset.json (레코드별 구조)
- **7. 품질 검증 —** 원문 근거·중복 제거·QA 품질·학습 적합성 평가
- **8. 버전 관리 —** version, created_by, record_count 메타데이터

### 5.3 데이터 계약(스키마)을 지켜라

파이프라인 단계 간 필드명·컬럼명을 임의로 바꾸면 다운스트림과 테스트가 깨진다. 표준 CSV 컬럼은 다음과 같다.

```
id, domain, category, question, answer, instruction, input, output, source_document, keyword, created_date
```

## 6. 인프라·하드웨어·도구 요건

### 6.1 하드웨어

| 구성 | 최소 | 권장 |
|------|------|------|
| 데이터 생성(추론만) | CPU + 16GB RAM (mock/API) | GPU 8GB+ 또는 외부 API |
| QLoRA 학습(소형) | GPU 6.5GB VRAM (9B 4bit) | RTX 30/40/50 24GB |
| LoRA 16bit 학습 | GPU 24GB | 다중 GPU / DGX 계열 |

**플랫폼.** Unsloth 학습은 NVIDIA(RTX 30~50, Blackwell, DGX), Intel GPU, Mac(MLX) 등에서 동작한다. GPU가 없으면 추론·Data Recipes만 가능하다. 자체 구축 시에도 데이터 생성은 CPU/API로, 학습은 GPU로 분리하는 것이 현실적이다.

### 6.2 소프트웨어 스택

- **언어/프레임워크 —** Python, FastAPI(웹 UI), PyTorch
- **추론 엔진 —** Ollama(로컬), vLLM(엔터프라이즈/다중 사용자), llama.cpp(GGUF)
- **학습 —** Unsloth(+Hugging Face transformers, TRL)
- **배포 포맷 —** GGUF(Ollama·LM Studio), safetensors, vLLM(FP8·AWQ)
- **폴백 설계 —** LLM 미가용 환경에서도 mock으로 파이프라인이 끝까지 돌도록(테스트 가능성)

### 6.3 배포·서빙 옵션

| 대상 | 포맷/엔진 | 비고 |
|------|-----------|------|
| 로컬·노트북·Mac | GGUF + llama.cpp/Ollama/LM Studio | 단일 장치 추론 |
| 엔터프라이즈·다중 사용자 | vLLM (FP8·AWQ) | 고처리량 서빙 |
| 도구 연동 | OpenAI 호환 API 엔드포인트 | Claude Code·Codex 등 |

## 7. 품질·보안·운영 고려사항

### 7.1 데이터 품질 게이트

- **원문 근거 검증 —** 생성 답변이 원문에 근거하는지 확인(환각 차단)
- **중복 제거 —** 유사/중복 레코드 제거로 과적합·편향 방지
- **LLM Judge —** QA 품질·학습 적합성을 모델로 점수화하고 임계값 게이트
- **검증셋 분리 —** 학습 데이터의 20%를 평가용으로 분리하거나 수동 평가

### 7.2 보안·라이선스·프라이버시

- **오프라인 우선 —** 민감 문서는 로컬 모델로 처리(외부 API 전송 금지 옵션)
- **라이선스 점검 —** Unsloth Studio의 AGPL-3.0 구성요소, 베이스 모델의 상업 사용 조건 확인
- **접근 통제 —** 토큰 기반 인증(JWT), 데이터셋·모델 산출물 권한 관리
- **PII 처리 —** 학습 데이터의 개인정보 마스킹·비식별화 절차

### 7.3 옵저버빌리티·버전 관리

학습 손실·그래디언트 노름·GPU 사용률을 실시간 추적하고, 데이터셋과 모델에 버전(version·created_by·record_count)을 부여해 재현성을 확보한다. 실험 이력을 저장해 재실행·비교가 가능해야 한다.

## 8. 구축 체크리스트와 로드맵

### 8.1 단계별 체크리스트

| 단계 | 할 일 | 완료 기준 |
|------|-------|-----------|
| 1. 데이터 | 로더·청킹·QA 생성·정제 구현 | 표준 스키마 산출, 검증 통과 |
| 2. 포맷 | CSV/JSON/JSONL 내보내기 | Unsloth/HF가 바로 로드 |
| 3. 모델 | 베이스 모델·방식 선정 | QLoRA로 소형부터 검증 |
| 4. 학습 | 하이퍼파라미터·학습 루프 | loss 0.5~1.0, 과적합 없음 |
| 5. 평가 | 검증셋·자동/수동 평가 | 기준 충족 |
| 6. 배포 | GGUF/vLLM·API 서빙 | 엔드포인트 동작 |
| 7. 운영 | 옵저버빌리티·버전·보안 | 재현성·접근통제 확보 |

### 8.2 권장 로드맵

1. **Now —** 데이터 생성 파이프라인 완성 + Unsloth JSONL 내보내기 추가 (이미 8할 구현)
2. **Next —** Unsloth 노트북/Studio로 소형 모델 QLoRA 파인튜닝 PoC, 베이스 vs 파인튜닝 비교
3. **Later —** RAG 백엔드(ChromaDB/FAISS) 연동, vLLM 서빙, 자동 평가·옵저버빌리티 내재화

> **결론**
> Unsloth는 학습을 쉽게 만드는 도구이고, 사내 시스템은 학습 데이터를 잘 만드는 도구다. 둘은 경쟁이 아니라 **상호보완** 관계다. 핵심 전략은 사내 데이터셋을 Unsloth가 바로 소비할 수 있는 포맷으로 내보내고, Studio의 Data Recipes·옵저버빌리티·Model Arena 같은 UX를 벤치마크해 우리 웹 UI를 고도화하는 것이다.
