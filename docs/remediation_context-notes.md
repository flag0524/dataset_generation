# 컨텍스트 노트 — 데이터셋 품질 개선 (법률 검증보고서 대응)

작성 2026-07-07. grill(/grill-me)로 도출한 결정과 근거.

## 근본 원인
보고서 §3·§5-1의 "초반 7개 청크만 실제 생성, 이후 폴백 템플릿"은 이번에 배포한 `LLM_TIME_BUDGET`(25초) 시간 예산이 직접 원인이다. 보고서 §3.1 템플릿 4종이 `pipeline._heuristic_output` 출력과 글자 단위로 일치한다. 즉 30초 SLA를 지키려고 예산 소진 후 세그먼트를 휴리스틱으로 폴백한 결과, 5,312건 중 약 99.5%가 문자열 조립본이 되어 파인튜닝에 부적합해졌다.

## 결정과 근거
1. **품질 우선, SLA 포기.** 데이터셋 생성은 파인튜닝용이라 모든 청크가 실제 LLM 생성이어야 의미가 있다. 30초 벽시계 상한과 "모든 청크 생성"은 문서당 1,300청크 규모에서 양립 불가. → 생성은 무제한, 30초는 생성 경로의 잘못된 제약이었다.
2. **폴백 없음(청크 제외).** 템플릿 폴백이 "진짜처럼 보이는 가짜"를 학습셋에 조용히 섞은 게 핵심 죄. LLM 답변이 없으면 그 레코드를 만들지 않는다. 총량은 줄지만 모든 레코드가 진짜.
3. **증강 제거.** min_rows(100) 패딩이 단일 소스를 패러프레이즈로 부풀려 148건 완전 중복과 불용어 키워드를 낳았다. 품질 우선과 모순이므로 제거. 크기 미달은 FAIL이 아니라 경고(소스 확대로 해결할 문제).
4. **이번 범위 = 파이프라인 수정 + 샘플 검증.** 무제한 생성은 CLI에서 자연스러움(HTTP 타임아웃 없음). 따라서 데이터셋 결함을 고치는 데 웹 백그라운드 잡이 필수는 아님. 웹 백그라운드·다중소스·법률 재생성은 후속.
5. **시간 예산 기본 OFF, 옵인 미리보기.** 예산 메커니즘은 삭제하지 않고 명시적 "빠른 미리보기" 모드로만 두되 산출물을 "미완성·학습용 아님"으로 라벨. 진짜 데이터셋을 다시는 조용히 오염하지 않게.
6. **원본 PDF 부재.** 재생성 소스 `2024590_국토교통위원회_검토보고서.pdf`가 레포에 없음(샘플은 `sample_admin.txt` 뿐). 따라서 법률 데이터셋 재생성은 PDF 확보 후 후속. 지금은 코드 수정 후 샘플로 검증.

## 예상 결과·주의
- 증강 제거로 `sample_admin.txt`(작은 문서)는 100행에 못 미친다. `test_t6_*`의 `row_count>=100`·`status==PASS`·`quality>=90` 단언은 새 의미(패딩 없음, 크기=경고)로 갱신해야 한다. TRD §5 / tests.md 검증 의미가 바뀌는 지점.
- 무제한 재생성은 실제 Ollama에서 문서 규모에 비례해 수십 분 걸릴 수 있다(품질 우선의 비용).
